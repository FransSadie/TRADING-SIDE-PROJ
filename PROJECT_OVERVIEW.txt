==================================================
TECHNICAL TERMS
==================================================

1) Purpose
- Build a trading support pipeline that ingests news + market data, transforms raw inputs into ML-ready features, and creates labels for prediction modeling.

2) Current Architecture
- Backend: FastAPI
- DB access: SQLAlchemy ORM
- DB target: PostgreSQL (URL normalized to postgresql+pg8000://), SQLite fallback supported
- Scheduling: APScheduler
- Data sources:
  - News: News API (/v2/everything)
  - Prices: Yahoo Finance via yfinance

3) Core Data Flow
- Ingestion layer:
  - News API articles -> news_articles
  - Yahoo OHLCV rows -> market_prices
  - Every job run logged -> ingestion_runs
- NLP layer:
  - Article text processed into per-article/per-ticker signals -> news_signals
- Feature layer:
  - Rolling aggregation by ticker/window -> feature_snapshots
- Label layer:
  - Future return + binary direction target -> market_labels
- Model layer:
  - Join features + labels by ticker/date
  - Time-ordered train/validation split
  - Logistic Regression baseline training
  - Save model and metadata artifacts
  - Predict endpoint uses latest feature row per ticker

4) Tables and Their Roles
- news_articles
  - Raw article payload storage (title, description, content, source, url, published_at, tickers)
  - Dedupe: unique(source_name, url)
- market_prices
  - OHLCV rows per ticker/timestamp/interval
  - Dedupe: unique(ticker, timestamp)
- ingestion_runs
  - Pipeline audit table for job_name, status, rows_inserted, start/end, message
- news_signals
  - NLP outputs per (article_id, ticker): sentiment_score, relevance_score
  - Dedupe: unique(article_id, ticker)
- feature_snapshots
  - Time-window aggregates for model inputs:
    news_count, sentiment_mean/sum/std, relevance_mean, price_close, return_1d
  - Dedupe: unique(ticker, window_end, window_hours)
- market_labels
  - Supervised targets: target_up + future_return by horizon_days
  - Dedupe: unique(ticker, timestamp, horizon_days)

5) Key Modules
- app/ingestion/news_api.py
  - News API client + response parsing
- app/ingestion/market_data.py
  - yfinance download + normalization
- app/ingestion/pipeline.py
  - News + market ingestion jobs
- app/nlp/pipeline.py
  - Ticker extraction + lexicon sentiment + relevance scoring
- app/features/pipeline.py
  - Feature aggregation + label generation
- app/api/routes.py
  - Endpoints for health, ingestion, and pipeline run/status
- app/db/session.py
  - DB engine/session config + postgres URL normalization
- app/models/dataset.py
  - Feature/label training dataset join + time split + latest feature lookup
- app/models/train_baseline.py
  - Baseline model training, metrics, artifact persistence
- app/models/inference.py
  - Artifact loading + per-ticker prediction

6) API Endpoints

- POST /ingest/run
- GET /ingest/status
- POST /pipeline/run
- GET /pipeline/status
- POST /model/train
- GET /model/status
- GET /predict?ticker=NVDA

7) How to Run
- Ingestion only:
  - python -m app.ingestion.run_once
- NLP + features + labels:
  - python -m app.features.run_once
- Train baseline model:
  - python -m app.models.train_baseline
- API server:
  - python -m uvicorn app.main:app --reload
- PowerShell helpers:
  - scripts/start_api.ps1
  - scripts/smoke_api.ps1
  - scripts/run_feature_pipeline.ps1
  - scripts/run_model_training.ps1

8) Model Artifacts
- Stored in MODEL_ARTIFACTS_DIR (default ./artifacts)
- baseline_model.joblib
  - Serialized scikit-learn LogisticRegression model
- baseline_metadata.json
  - Feature columns, train/validation metrics, creation timestamp
  - Includes train_classes and model_type; can be dummy_most_frequent when training data has one class

9) Current State
- End-to-end flow is implemented and runnable.
- Data has been successfully written to PostgreSQL.
- Feature/label generation works after ingestion data is available.
- Baseline model training and prediction endpoints are implemented.
- Current training data is small/imbalanced, so baseline may use DummyClassifier fallback until more varied data is collected.

10) Next Technical Milestone
- Improve model quality and robustness:
  - richer NLP/entity mapping
  - more features (volume spikes, lagged returns, rolling volatility)
  - larger historical data window
  - walk-forward validation/backtest
  - risk controls for signal consumption

==================================================
VERY SIMPLE TERMS
==================================================

What this project does right now:
- It fetches news.
- It fetches stock prices.
- It saves both in your database.
- It keeps a log of each run so you can see if it worked.
- It reads news text and gives each article a simple score (good/bad and how relevant it is).
- It groups those scores into clean rows for AI training.
- It creates the correct answers from real future prices (did price go up or down?).
- It trains a first basic model from that prepared data.
- It can give a prediction (up/down + confidence) for a ticker using latest features.

Think of it like a factory line:
1) Collect raw material (news + prices)
2) Clean and score it
3) Pack it into training data
4) Prepare answer keys for learning

What is NOT done yet:
- Advanced model quality and full trading strategy validation are not done yet.

Next simple step:
- Improve the model with more data and better features so predictions are more useful.
